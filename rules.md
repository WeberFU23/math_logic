# 规则驱动 Agent 架构与规则说明

## 架构总览

该 Agent 采用**分层符号化架构**，将像素观测转换为符号状态，再通过优先级链决策。核心模块及数据流如下：

```
像素帧 (obs)  ──► vision.py + color_vision.py  ──► SymbolicState
                                                         │
                                           ┌─────────────┘
                                           ▼
                              strategy.py (选择 Goal)
                                           │
                                           ▼
                              planner.py  (BFS 寻路)
                                           │
                                           ▼
                              executor.py (Goal → 动作)
                                           │
                                           ▼
                              shield.py   (安全检查)
                                           │
                                           ▼
                              agent.py    (战斗反射、卡墙处理等)
```

---

## 1. 视觉感知层 — `vision.py` + `color_vision.py`

### 1.1 颜色模式自适应 (`color_vision.py`)

每一帧 Agent 都会获得标准的 NumPy uint8 RGB 数组作为当前输入。Agent会根据第一帧**自动推断**评测框架施加的颜色变换（灰度/暗色/亮色/高对比度/反色），无需 `info`：

- `high_contrast`：所有像素值仅为 0 或 255
- `grayscale`：R == G == B
- `dark`：最大像素值 ≤ 141
- 否则比较 `default` / `bright` / `inverted` 三种模式下 floor 颜色的 MSE 误差，取误差最小的模式

随后对模板库施加相同颜色变换（`transform_color_image`），实现模板匹配在任意颜色变体下有效。

### 1.2 静态瓦片分类 (`vision.py`)

对 10×8 网格的每个 16×16 瓦片，使用**预渲染模板库**进行 MSE（均方误差）匹配，分类为：

| 标签 | 含义 |
|---|---|
| `wall` | 墙壁 |
| `chest` | 宝箱（合并 key/gold/heal 三种） |
| `trap` / `abyss` | 陷阱 / 深渊 |
| `button` / `button_pressed` | 按钮 / 已按下按钮 |
| `switch` / `switch_pressed` | 开关 / 已激活开关 |
| `gap` / `bridge` | 深渊 / 桥 |
| `exit_normal` / `exit_locked_key` / `exit_conditional` | 普通出口 / 锁门 / 条件门 |
| `npc` | NPC |
| `floor` | 地板 |
| `unknown` | 无法识别 |

**关键规则**：

- **多背景模板**：对不填满瓦片的对象（宝箱、陷阱、按钮、开关、NPC），分别在 `floor` / `bridge` / `gap` 三种背景上生成模板，保证任意地形下都能正确匹配
- **位置感知模板**：出口瓦片根据其在地图边界的具体位置（北/南/西/东 + 左/右瓦片）生成不同的模板，区分不同类型的门
- **时序记忆** (`_merge_memory`)：当瓦片被动态精灵遮挡超过 35% 或被分类为 `unknown` 时，回退到上一帧的分类结果，弥合视觉遮挡

### 1.3 动态精灵检测 (`color_vision.py`)

玩家和怪物通过**精灵模板卷积**检测：

- **玩家检测**：
  - 使用 4 方向精灵模板（上/下/左/右），配合玩家专属调色板（`PLAYER_PALETTE`：`OUTLINE`, `PLAYER_TUNIC`, `PLAYER_TUNIC_LIGHT`, `PLAYER_FACE`, `PLAYER_HAIR`）
  - 在前一帧位置 ±4px 范围内局部搜索
  - 若局部搜索得分 > 5000（可能丢失），回退到全图搜索

- **怪物检测**：
  - 3 种类型（`chaser` / `patroller` / `ambusher`），各自独立的调色板
  - 已有跟踪的怪物：仅在上一帧 origin ±4px 范围内局部跟踪
  - 首次检测或丢失目标时：全图扫描（`_scan_monsters`）
  - 丢失后 8 帧内持续全图重扫描（`monster_rescan_ticks`）

- **去重**：NMS（非极大值抑制），半径 8px 内保留得分最低的匹配

- **边界稳定性** (`_stabilize_boundary_anchor`)：当精灵中心恰好落在瓦片边界上时（坐标 % TILE_SIZE == 0），根据上一帧的运动方向决定归属瓦片

## 2. 符号状态 — `symbolic.py`

定义了核心数据结构：

| 类型 | 含义 |
|---|---|
| `Position = (int, int)` | 10×8 网格中的瓦片坐标 |
| `RoomCoord = (int, int)` | 房间坐标（Agent 自行推断，不以环境 info 为准） |
| `GlobalPosition = (int, int, int, int)` | 全局坐标 `(room_x, room_y, tile_x, tile_y)` |
| `SymbolicState` | 一帧的完整符号快照（见下方字段） |
| `AgentMemory` | 跨帧持久记忆（见下方字段） |
| `Goal` | 目标 `(GoalKind, target_position)` |

**7 种动作常量**：

| 常量 | 值 | 含义 |
|---|---|---|
| `ACTION_NOOP` | 0 | 无操作 |
| `ACTION_UP` | 1 | 向上移动 |
| `ACTION_DOWN` | 2 | 向下移动 |
| `ACTION_LEFT` | 3 | 向左移动 |
| `ACTION_RIGHT` | 4 | 向右移动 |
| `ACTION_A` | 5 | 使用剑（攻击） |
| `ACTION_B` | 6 | 使用盾（防御） |

**6 种目标类型** (`GoalKind`)：

| 类型 | 含义 |
|---|---|
| `ATTACK_MONSTER` | 攻击怪物 |
| `OPEN_CHEST` | 开启宝箱 |
| `ACTIVATE_SWITCH` | 激活开关（站旁边按 A） |
| `ACTIVATE_BUTTON` | 激活按钮（站上去） |
| `GO_TO_EXIT` | 前往出口 |
| `EXPLORE` | 探索未知区域 |
| `WAIT` | 等待 |

### `SymbolicState` 字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `player` | `Position` | 玩家瓦片坐标 |
| `player_center_px` | `(float, float)` | 玩家精灵中心像素坐标（用于对齐微调） |
| `room` | `RoomCoord` | 当前房间坐标（从 memory 继承） |
| `walls` | `set[Position]` | 墙壁瓦片 |
| `chests` | `set[Position]` | 宝箱瓦片 |
| `monsters` | `set[Position]` | 怪物瓦片 |
| `normal_exits` | `set[Position]` | 普通出口 |
| `locked_exits` | `set[Position]` | 锁住的出口 |
| `conditional_exits` | `set[Position]` | 条件出口 |
| `traps` | `set[Position]` | 陷阱/深渊 |
| `buttons` | `set[Position]` | 按钮 |
| `switches` | `set[Position]` | 开关 |
| `bridges` | `set[Position]` | 桥 |
| `gaps` | `set[Position]` | 深渊（减去桥） |
| `npcs` | `set[Position]` | NPC |
| `keys` | `int` | 持有钥匙数 |
| `gold` | `int` | 持有金币数 |
| `last_reward` | `float` | 上一步奖励 |
| `health` | `int \| None` | 血量（仅视觉可推断时） |
| `has_sword` | `bool` | 是否有剑 |
| `has_shield` | `bool` | 是否有盾 |

### `AgentMemory` 字段

| 字段 | 含义 |
|---|---|
| `task_id` | 当前任务 ID |
| `room` | 当前房间坐标 |
| `last_goal` | 上一步的目标 |
| `opened_chests` | 全局已开启宝箱集合 |
| `activated_buttons` | 全局已激活的按钮集合 |
| `activated_switches` | 全局已激活的开关集合 |
| `used_exits` | 全局已使用的出口集合 |
| `previous_monsters` | 上一帧怪物位置 |
| `previous_player` | 上一帧玩家位置 |
| `last_action` | 上一步执行的动作 |
| `room_steps` | 当前房间已执行步数 |
| `switch_button_cooldown` | 开关/按钮冷却计数器（对两者生效） |

**房间切换检测** (`observe_room_transition`)：当玩家从边界消失并在对面出现时推断穿越方向。例如从 `col ≤ 1` 跳到 `col ≥ 7` 且动作为 LEFT → 向西移动一个房间。穿越后自动标记出发和到达出口为"已使用"。

**宝箱开启检测**：当动作 A 面朝宝箱目标，且观察到钥匙/金币增加或奖励 > 0 或获得剑时，标记该宝箱为已开启。

**按钮检测**：玩家站在按钮瓦片上时自动标记为已激活。

## 3. 策略层（优先级链）— `strategy.py`

`RuleBasedPolicy.choose_goal()` 定义了一条**12 级固定优先级链**：

| 优先级 | 目标 | 触发条件 |
|---|---|---|
| **1. STICKY** | 继续上一帧的目标 | 目标仍然存在且可达（详见 §3.1） |
| **2. CHEST** | 前往最近的可达宝箱 | 存在未开启的可达宝箱 |
| **3. MONSTER** | 清理房间怪物 | 可达怪物存在 ∧ 有剑 ∧ 血量 > 1 ∧ 房间无宝箱/开关/按钮/未使用出口（房间已搜刮干净） |
| **4. COND_DOOR** | 处理条件门（三选一） | 房间有条件门 |
| ↳ 4a. BUTTON | 按下按钮 | 按钮未全按，存在可达按钮 |
| ↳ 4b. MONSTER | 击杀守门怪物 | 无按钮、有活怪、有剑、血量 > 1、可达 |
| ↳ 4c. EXIT | 走未使用的条件出口 | 前提满足（按钮全按或怪物全清），存在可达新出口 |
| **5. EXIT_NEW_L** | 走未使用的锁门 | 有钥匙，存在可达的未使用锁门 |
| **6. EXIT_NEW_N** | 走未使用的普通门 | 存在可达的未使用普通门 |
| **7. SWITCH** | 激活开关 | 存在可达开关 ∧ 不在冷却期（优先未激活的，回退到已激活的） |
| **8. EXIT_USED_N** | 走已使用的普通门（回退） | 存在可达的普通门（含已使用） |
| **9. COND_DOOR_USED** | 条件门回退（三选一） | 房间有条件门且新出口不可用 |
| ↳ 9a. BUTTON | 按下按钮 | 同 4a |
| ↳ 9b. MONSTER | 击杀守门怪物 | 同 4b |
| ↳ 9c. EXIT | 走条件出口（含已使用） | 同 4c，但允许已使用的出口 |
| **10. EXIT_USED_L** | 走已使用的锁门（回退） | 有钥匙，存在可达的锁门（含已使用） |
| **11. EXPLORE** | 探索 | BFS 找到最近的未阻塞瓦片 |
| **12. WAIT** | 等待 | 以上全部不满足 |

### 3.1 Sticky 目标规则 (`_continue_last_goal`)

上一帧目标在以下条件继续有效：

- **宝箱**：目标宝箱仍未被开启，且可达（或距离 = 1）
- **怪物**：目标怪物仍存活，有剑且血量 > 1，且可达（或距离 = 1）
- **按钮**：目标按钮仍未被激活，且可达；若玩家已站在上面则放弃
- **开关**：若存在未使用的出口则放弃开关（优先离开）；否则继续
- **出口**：
  - 若存在未开启宝箱 → 放弃出口
  - 若是锁门且无钥匙 → 放弃
  - 若是条件门且前提未满足 → 放弃
  - 若需清理怪物 → 放弃
  - 若已在出口上 → 仅当该出口全局未使用时继续

### 3.2 怪物清理条件 (`_must_clear_monsters_before_exit`)

只有当房间中**同时满足以下全部条件**才战斗：

```
有剑 ∧ 有怪物 ∧ 血量 > 1 ∧ 无宝箱 ∧ 无开关 ∧ 无按钮 ∧ 无未使用的出口
```

即"房间已搜刮干净才战斗"原则。

### 3.3 出口排名规则 (`_best_exit_goal`)

按五元组排序，优先级从高到低：

```
(是否已使用, 前方是否被阻挡, 是否入口侧, BFS距离, 坐标)
```

- **优先未使用**的出口
- **优先前方无阻挡**的出口（出口前方一格无墙壁/陷阱/深渊/宝箱/怪物/NPC）
- **避免入口侧**：进入房间 ≤ 4 步内，避免立即从进来的门返回
- 同等条件下选 BFS 距离最近的

## 4. 寻路层 — `planner.py`

### 4.1 可行走判断 (`is_walkable`)

一个瓦片**不可行走**当且仅当：

```
瓦片 ∈ 墙壁 ∪ 陷阱 ∪ 宝箱 ∪ 怪物 ∪ NPC ∪ (深渊 − 桥)
```

若 `allow_goal=True`，出口瓦片视为可行走（允许玩家站在出口瓦片上）。

### 4.2 BFS 寻路 (`bfs_path`)

- 标准 BFS，四方向移动
- 支持 `preferred_action`：优先尝试保持当前移动方向（减少蛇形移动）
- 目标瓦片（出口等）使用 `allow_goal=True` 判断可行走性

### 4.3 目标瓦片映射 (`goal_tiles`)

将 Goal 转换为 BFS 的目标瓦片集合：

| Goal 类型 | 目标瓦片集合 |
|---|---|
| `OPEN_CHEST` / `ATTACK_MONSTER` | 目标相邻四方向的可行走瓦片 |
| `ACTIVATE_BUTTON` | 按钮瓦片本身（站上去触发） |
| `ACTIVATE_SWITCH` | 开关相邻可行走瓦片（站旁边按 A） |
| `GO_TO_EXIT` | 出口瓦片本身 |
| `EXPLORE` | 目标瓦片本身 |

## 5. 执行层 — `executor.py`

`action_for_goal(state, goal)` 将 Goal 转换为具体动作：

| Goal 类型 | 动作逻辑 |
|---|---|
| `WAIT` | 返回 `NOOP` |
| `OPEN_CHEST` / `ATTACK_MONSTER` | 距离 = 1 → 按 A（剑）；否则 BFS 移动一步 |
| `ACTIVATE_BUTTON` | 已在上面 → `NOOP`；否则 BFS 走向按钮 |
| `ACTIVATE_SWITCH` | 距离 = 1 → 按 A；否则 BFS 移动一步 |
| `GO_TO_EXIT` | 已在出口上 → 向外推（上/下/左/右根据出口位置）；否则 BFS 移动一步 |
| `EXPLORE` | BFS 移动一步 |

**出口推出动作** (`_exit_push_action`)：根据出口瓦片位置决定推出方向——row=0 向上推，row=7 向下推，col=0 向左推，col=9 向右推。

---

## 6. 安全盾 — `shield.py`

`shield(action, state)` 是最外层的安全检查：

- `NOOP`、`A`、`B` 动作直接放行
- 非移动动作替换为 `NOOP`
- 移动动作：若目标瓦片不可行走（且不是站在出口上向外推出），替换为 `NOOP`
- 站在出口上向外推出时跳过检查（允许离开地图边界）

## 7. 主循环 — `agent.py`

`Policy.act()` 是核心决策循环，融合了多层机制。每帧处理流程：

```
1. 视觉感知 → SymbolicState
2. 运动检测（卡墙判断）
3. memory.update()（状态追踪、房间切换检测）
4. 房间切换 → 清空队列和战斗状态
5. 新怪物检测（穿越出口时）→ 中断队列，重新规划
6. 队列执行（如有排队 tick）
   ├── 检查战斗反射 → 中断队列
   └── 否则发出排队动作
7. 无队列时：
   a. 战斗反射检查（最高优先级）
   b. 强制战斗目标（锁定追击）
   c. 警戒战斗目标（主动迎击）
   d. 高层策略选择 Goal
   e. Goal → 动作
   f. 卡墙处理（对齐微调 / 脱困动作）
   g. 安全盾检查
   h. 动作入队
```

### 7.1 动作队列

移动动作不是逐帧发出的，而是一次性入队多个 tick：

| 场景 | 入队 Ticks |
|---|---|
| 无怪物 | 15（走一整格 16px，略少于精确值以留余量） |
| 怪物距离 ≤ 2 | 1（逐像素移动，便于随时插入战斗反射） |
| 怪物距离 > 2 | 3 |
| 穿越出口 | 23（跨房间加载需要更多帧） |
| 像素微调（对齐 nudges） | 0（不入队，单帧执行） |

### 7.2 战斗反射 (`_combat_reflex`)

**在所有队列执行期间都检查**，优先级最高。仅当怪物在玩家 8 邻域内（切比雪夫距离 = 1）且无墙体阻隔时触发：

**对角线相邻怪物**（dx 和 dy 均非零，即切比雪夫距离 = 1 但曼哈顿距离 = 2）：

1. 首次检测到该怪物：有盾则举盾 B，持续最多 `_DIAGONAL_BLOCK_TICKS`（2 帧）
2. 格挡结束后：有剑则转向怪物并挥剑 A
3. 无剑有盾：继续举盾 B
4. 无剑无盾：后退远离

**正交相邻怪物**（曼哈顿距离 = 1）：

1. 有剑且已面朝怪物 → 挥剑 A（若连续挥剑则交替举盾 B 防御）
2. 有剑但未面朝 → 有盾则先举盾 B；否则转向怪物（移动可能被阻挡但朝向会更新）
3. 有盾无剑 → 举盾 B
4. 无剑无盾 → 后退一步

**锁定目标不在近身**（其他怪物靠近）→ 举盾 B 防御，并后退远离新威胁，继续追击锁定目标。

### 7.3 战斗锁定 (`_combat_target`)

- 首次攻击怪物时锁定目标，持续追踪其瓦片位置变化
- 锁定持续 **64 ticks**（`_FIGHT_COMMIT_TICKS`），防止在怪物间频繁切换
- 怪物消失或数量减少 → 解锁（可能已被击杀）
- 怪物位置变化时通过最小化曼哈顿距离追踪同一目标

### 7.4 警戒战斗 (`_alert_combat_goal`)

在怪物进入**曼哈顿距离 ≤ 2** 且无墙体阻隔时主动迎击（不等它贴脸），条件是有剑且血量 > 1。

### 7.5 卡墙检测 (`_observe_motion`)

**纯视觉方式检测**（不使用 `info`）：

1. 记录连续两帧玩家像素中心的位移
2. 若朝某方向移动但沿该方向的位移 < 0.5px，累计静止帧数
3. 累计 ≥ 2 帧后判定为被阻挡
4. 记录被阻挡的动作和方向，触发后续**脱困机制**
5. 最大连续检测 6 帧后重置

房间切换或位移 ≥ 0.5px 时清零计数器。

### 7.6 对齐微调 (`_alignment_action`)

当玩家像素中心偏离瓦片中心超过阈值时，发出垂直于移动方向的微调动作。

**触发条件**：玩家被阻挡（`_blocked_ticks > 0`）且移动方向前方有狭窄通道。

**水平移动（LEFT/RIGHT）时**：
- 前方上下两侧均被阻挡（如走廊）→ 偏离中心 > 2px 时向中心微调
- 仅上方被阻挡 → 偏上时向下微调（远离墙壁）
- 仅下方被阻挡 → 偏下时向上微调（远离墙壁）

**垂直移动（UP/DOWN）时**：同理，检查左右两侧。

### 7.7 脱困动作 (`_unstick_action`)

移动被阻挡且对齐微调不适用时的回退策略：

1. **出口角落绕行** (`_exit_corner_nudge`)：若目标是边界出口且前方角落被阻挡，沿边界垂直方向绕行
2. **通用脱困**：尝试其他三个方向，选择：
   - 可行走
   - 不走向怪物（距离 > 1）
   - 最接近目标的瓦片
   - 优先同轴方向（减少蛇形）

### 7.8 出口对齐 (`_exit_alignment_action`)

到达出口瓦片但像素未居中对齐门洞时的微调：

- 北/南出口（row = 0 或 7，col ∈ {4, 5}）：调整玩家水平位置（左右微调）
- 西/东出口（col = 0 或 9，row ∈ {3, 4}）：调整玩家垂直位置（上下微调）
- 容差 ±1.5px

### 7.9 穿越出口中断

队列执行期间若正在穿越出口且检测到新怪物出现，立即清空队列并重新规划退出目标，防止盲目走进有怪物的房间。

## 8. 规则设计哲学

这套规则系统的核心设计哲学是：**将塞尔达式迷宫的所有交互抽象为符号化的 Goal 优先级链，结合像素级视觉感知、BFS 空间规划、战斗反射和卡墙恢复，完全不依赖环境内部 `info` 状态**。

核心优势：

1. **颜色自适应**使同一套模板在 5 种颜色变体下都能工作
2. **优先级链**保证了任务链的顺序正确性（先搜刮 → 再战斗 → 最后离开）
3. **战斗反射**在所有层级之上，保证生存
4. **卡墙检测 + 对齐 + 脱困**三层机制处理像素级移动的不确定性
5. **时序记忆**弥合视觉遮挡和跨帧信息丢失
6. **模块化设计**：`HighLevelPolicy` 抽象基类允许未来用 RL 策略替换 `RuleBasedPolicy`，不影响感知、规划、执行和安全盾层
